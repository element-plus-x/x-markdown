# 组件库动态引入测试流程

## 概述

本文档描述了测试 x-markdown 组件库可选依赖（shiki、shiki-stream、mermaid）降级功能的完整流程。

## 目标

确保当可选依赖未安装时，组件库能够正确降级为纯文本模式，不会产生任何错误。

## 测试流程步骤

### 1. 修改组件库代码

在 `x-markdown/packages/x-markdown/src/hooks/` 目录下修改以下文件：
- `useHighlight.ts` - 处理 shiki 和 shiki-stream 的动态导入
- `useMermaid.ts` - 处理 mermaid 的动态导入

**关键实现**：使用 `Function` 构造器绕过 Vite 的静态分析

```typescript
// 使用动态导入辅助函数来绕过 Vite 的静态分析
const dynamicImport = async (moduleName: string): Promise<any> => {
  const importFn = new Function('moduleName', `return import(moduleName)`)
  try {
    return await importFn(moduleName)
  } catch {
    return null
  }
}

// 使用示例
const mod = await dynamicImport('shiki')
if (!mod) {
  // 降级逻辑
  showDependencyHint()
  return null
}
```

## 常见问题

### 问题1: Vite 预构建失败

**错误信息**: `Failed to resolve import "shiki" from "node_modules/.vite/deps/xxx.js"`

**解决方案**:
1. 在测试项目的 `vite.config.js` 中添加 `optimizeDeps.exclude`
2. 清理 Vite 缓存：`rm -rf node_modules/.vite`
3. 重启开发服务器

### 问题2: npm publish 需要 OTP

**错误信息**: `This operation requires a one-time password`

**解决方案**: 从你的身份验证器应用获取 OTP 码并输入

### 问题3: 包入口点错误

**错误信息**: `Failed to resolve entry for package`

**解决方案**: 检查 `package.json` 中的入口点配置是否与实际构建输出匹配：

```json
{
  "main": "./dist/x-markdown.cjs.js",
  "module": "./dist/x-markdown.es.js",
  "types": "./dist/types/index.d.ts"
}
```

## 技术要点

### 核心原理

**问题**：Vite 在构建时会静态分析所有的 `import()` 语句，即使这些导入被包裹在 try-catch 中，Vite 仍会在预构建阶段尝试解析这些模块。

**解决方案**：使用 JavaScript 的 `Function` 构造器在运行时动态执行导入，完全绕过 Vite 的静态分析。

```typescript
// ❌ 会被 Vite 静态分析
const mod = await import('shiki')

// ✅ 绕过静态分析
const dynamicImport = async (moduleName: string) => {
  const importFn = new Function('moduleName', `return import(moduleName)`)
  return await importFn(moduleName)
}
const mod = await dynamicImport('shiki')
```

### Vite 配置要点

1. **external 配置**：在 `vite.config.ts` 中将可选依赖标记为 external
2. **optimizeDeps.exclude**：在测试项目中排除可选依赖的预构建
3. **preserveModules**：保持模块结构，便于调试
