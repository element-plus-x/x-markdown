# 动态导入与降级实现的技术要点

## 问题背景

在开发 x-markdown Vue 组件库时，需要支持可选依赖（shiki、shiki-stream、mermaid）的优雅降级：
- 当这些依赖**已安装**时，提供完整的语法高亮和图表渲染功能
- 当这些依赖**未安装**时，降级为纯文本模式，不产生任何错误

## 核心挑战

### 1. Vite 的静态分析问题

**现象**：
即使使用 `try-catch` 包裹动态导入语句，Vite 仍会在构建时解析这些模块，导致消费方项目在预构建阶段失败。

```typescript
// ❌ 这种写法会被 Vite 静态分析
try {
  const mod = await import('shiki')
  // 使用 mod
} catch {
  // 降级逻辑
}
```

**问题**：
- Vite 在预构建阶段会扫描所有的 `import()` 语句
- 即使在 try-catch 中，Vite 仍会尝试解析这些模块
- 如果模块不存在，整个构建过程会失败

### 2. peerDependencies 的自动安装问题

**现象**：
即使将可选依赖标记为 `peerDependencies` 并设置 `peerDependenciesMeta.optional: true`，pnpm 仍会自动安装这些依赖。

**原因**：
pnpm 的默认行为是自动安装 peer dependencies，即使标记为可选。

## 解决方案

### 方案1: 移除 peerDependencies（已采用）

**实施**：
- 从 `package.json` 的 `peerDependencies` 中完全移除 shiki、shiki-stream、mermaid
- 不在任何依赖列表中声明这些包
- 在代码中使用动态导入

**效果**：
- pnpm 不会自动安装这些包
- 用户可以自行决定是否安装

### 方案2: Function 构造器绕过静态分析（关键突破）

**实施**：
```typescript
const dynamicImport = async (moduleName: string): Promise<any> => {
  const importFn = new Function('moduleName', `return import(moduleName)`)
  try {
    return await importFn(moduleName)
  } catch {
    return null
  }
}

// 使用
const mod = await dynamicImport('shiki')
if (!mod) {
  // 降级逻辑
  return
}
```

**原理**：
- `Function` 构造器在运行时动态创建函数
- Vite 的静态分析无法识别字符串中的 `import()` 调用
- 只有在代码实际执行时才会尝试导入模块

**对比**：

| 方法 | Vite 静态分析 | 运行时导入 | 降级支持 |
|------|--------------|------------|----------|
| `import('shiki')` | ✅ 会被分析 | ✅ | ❌ 构建时失败 |
| `new Function()` | ❌ 不会被分析 | ✅ | ✅ 完美支持 |

## 实现细节

### useHighlight.ts 中的实现

```typescript
// 使用动态导入辅助函数来绕过 Vite 的静态分析
const dynamicImport = async (moduleName: string): Promise<any> => {
  const importFn = new Function('moduleName', `return import(moduleName)`)
  try {
    return await importFn(moduleName)
  } catch {
    return null
  }
}

let shikiModulePromise: Promise<any | null> | null = null
let shikiStreamModulePromise: Promise<any | null> | null = null

const loadShiki = async () => {
  if (!shikiModulePromise) {
    shikiModulePromise = (async () => {
      try {
        const mod = await dynamicImport('shiki')
        if (!mod) {
          showDependencyHint()
        }
        return mod
      } catch {
        showDependencyHint()
        return null
      }
    })()
  }
  return shikiModulePromise
}

const loadShikiStream = async () => {
  if (!shikiStreamModulePromise) {
    shikiStreamModulePromise = (async () => {
      try {
        const mod = await dynamicImport('shiki-stream')
        if (!mod) {
          showDependencyHint()
        }
        return mod
      } catch {
        showDependencyHint()
        return null
      }
    })()
  }
  return shikiStreamModulePromise
}

const showDependencyHint = () => {
  if (hasShownDependencyHint) return
  hasShownDependencyHint = true

  console.log(
    '%c[x-markdown]%c 代码高亮功能已降级为纯文本模式',
    'font-weight: bold; color: #0066cc;',
    'color: #666;'
  )
  console.log(
    '%c如需语法高亮功能，请安装以下依赖：',
    'color: #666; font-weight: bold;'
  )
  console.log(
    '%c  pnpm add shiki shiki-stream',
    'color: #00aa00; font-family: monospace;'
  )
}
```

### useMermaid.ts 中的实现

```typescript
const dynamicImport = async (moduleName: string): Promise<any> => {
  const importFn = new Function('moduleName', `return import(moduleName)`)
  try {
    return await importFn(moduleName)
  } catch {
    return null
  }
}

let mermaidPromise: Promise<any> | null = null
let mermaidAvailableCache: boolean | null = null
let mermaidCheckPromise: Promise<boolean> | null = null

export async function checkMermaidAvailable(): Promise<boolean> {
  if (mermaidAvailableCache !== null) {
    return mermaidAvailableCache
  }

  if (mermaidCheckPromise) {
    return mermaidCheckPromise
  }

  mermaidCheckPromise = (async () => {
    try {
      const mod = await dynamicImport('mermaid')
      mermaidAvailableCache = !!mod
      return mermaidAvailableCache
    } catch (error) {
      mermaidAvailableCache = false
      return false
    }
  })()

  return mermaidCheckPromise
}

async function loadMermaid() {
  if (typeof window === 'undefined') return null
  if (!mermaidPromise) {
    mermaidPromise = (async () => {
      try {
        const mod = await dynamicImport('mermaid')
        if (!mod) {
          showMermaidHint()
          return null
        }
        return (mod as any).default
      } catch {
        showMermaidHint()
        return null
      }
    })()
  }
  return mermaidPromise
}
```

## 为什么 external 不够用

### Rollup 的 external 限制

Rollup 的 `external` 配置主要用于：
- ✅ 防止将模块打包到输出中
- ✅ 保持 import 语句不变

但对于动态导入：
- ❌ 无法阻止 Vite 在预构建阶段解析 `import()` 语句
- ❌ 无法绕过静态分析

### 构建输出对比

**使用 `import()` 直接导入**：
```javascript
// 构建后的代码
const mod = await import("shiki"); // Vite 会尝试预构建这个
```

**使用 Function 构造器**：
```javascript
// 构建后的代码
const dynamicImport = async (moduleName) => {
  const importFn = new Function("moduleName", `return import(moduleName)`);
  return await importFn(moduleName);
};
const mod = await dynamicImport("shiki"); // Vite 无法分析
```

## 关键经验总结

### 1. 理解构建工具的静态分析行为

**要点**：
- Vite/Rollup 会在构建时分析所有的 `import()` 语句
- 即使包裹在 try-catch 中，仍会在预构建阶段尝试解析
- `external` 配置不能阻止这种静态分析

### 2. Function 构造器的巧妙运用

**优势**：
- 完全绕过静态分析
- 保留动态导入的运行时特性
- try-catch 能够正常工作

**注意**：
- 代码安全性：字符串中的代码在运行时执行
- TypeScript 类型：需要适当的类型标注

### 3. 降级设计的最佳实践

**原则**：
1. **优雅降级**：缺少依赖时不应该崩溃
2. **用户提示**：明确告知用户如何启用完整功能
3. **一次提示**：避免控制台被重复信息刷屏
4. **功能缓存**：避免重复检测依赖

**实现模式**：
```typescript
// 1. 模块级别的 Promise 缓存
let modulePromise: Promise<any> | null = null

// 2. 提示信息的标志位
let hasShownHint = false

// 3. 可用性缓存
let availableCache: boolean | null = null

// 4. 统一的加载函数
const loadModule = async () => {
  if (!modulePromise) {
    modulePromise = (async () => {
      const mod = await dynamicImport('module-name')
      if (!mod) {
        showHint()
      }
      return mod
    })()
  }
  return modulePromise
}
```

### 4. pnpm 的依赖管理特点

**关键理解**：
- pnpm 会自动安装 peer dependencies（即使标记为 optional）
- 如果希望完全可选，需要从 peerDependencies 中移除
- 使用动态导入时，不需要在任何依赖列表中声明

### 5. Vite 的预构建行为

**理解预构建**：
- Vite 会在启动时预构建依赖
- 扫描所有的 `import()` 语句
- 尝试解析并优化这些导入

**绕过方法**：
- 使用 `optimizeDeps.exclude` 排除特定依赖
- 使用 Function 构造器绕过静态分析
- 使用 `--force` 强制重新预构建

## 潜在风险与注意事项

### 1. 安全性考虑

**风险**：Function 构造器执行任意字符串代码

**缓解措施**：
- 仅用于已知的模块名
- 不接受用户输入作为模块名
- 代码审查确保使用安全

### 2. TypeScript 类型支持

**问题**：动态导入导致类型推断困难

**解决方案**：
```typescript
const loadMermaid = async () => {
  const mod = await dynamicImport('mermaid')
  // 类型断言
  return (mod as any)?.default
}
```

### 3. 调试难度

**挑战**：动态导入的错误堆栈可能不够清晰

**建议**：
- 添加详细的错误日志
- 使用有意义的变量名
- 在 catch 块中记录上下文信息
